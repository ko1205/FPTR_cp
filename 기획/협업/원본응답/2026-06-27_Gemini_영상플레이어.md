### 1. Frame-Accurate Stepping in HTML5 `<video>`
* **Cleanest Approach:**
  * **FPS Calculation:** In the `loadedmetadata` handler, calculate dynamically: `const fps = version.frame_count / video.duration`. Fallback to a hardcoded standard (e.g., `24` or `29.97`) if properties are invalid or missing.
  * **Stepping Formula:** Set video to pause, then target a specific discrete frame index to avoid drift:
    ```javascript
    const currentFrame = Math.round(video.currentTime * fps);
    const targetFrame = Math.max(0, Math.min(version.frame_count - 1, currentFrame + direction));
    video.currentTime = targetFrame / fps;
    ```
  * **Usage of `requestVideoFrameCallback` (rvfc):** Use rvfc only for synchronizing overlays or metadata on active play/render loops. For *stepping while paused*, modifying `currentTime` directly is required.
* **Top Pitfalls:**
  * **Interframe (Long-GOP) Lag:** Standard MP4 H.264 has keyframes (I-frames) only every few seconds. Stepping to P/B frames requires the browser to decode preceding frames on the fly, leading to seek lag, visual stutter, or temporary freezing. *Fix: MP4 files must be encoded with intra-only compression (e.g., keyframe interval `g = 1` via ffmpeg).*
  * **Floating-Point Drift:** Adding `currentTime += 1/fps` iteratively causes rounding errors, eventually landing between frame boundaries. Always anchor changes to `Math.round(currentTime * fps)` or a tracked integer `currentFrameIndex`.

### 2. Player Structure in React
* **Cleanest Approach:**
  * **Refs vs. State:** Store the video element reference using `useRef`. Maintain a single React state `isPlaying` (boolean) to toggle Play/Pause buttons.
  * **Avoiding Re-renders:** Do not store `currentTime` in React state. Instead, use local DOM operations inside `timeupdate` and `seeked` event listeners to update the frame counter text content directly:
    ```javascript
    const handleTimeUpdate = () => {
      if (counterRef.current) {
        const frame = Math.round(videoRef.current.currentTime * fps);
        counterRef.current.textContent = String(frame);
      }
    };
    ```
  * **Essential Events:** Listen to `loadedmetadata` (calculate FPS), `play`/`pause` (sync UI state), `timeupdate`/`seeked` (update scrubber/frame text), and `ended` (flip `isPlaying` state to false).
* **Top Pitfalls:**
  * **React Re-render Storm:** Hooking up `timeupdate` to React state causes the entire player layout to re-render 30–60 times a second, dragging down UI responsiveness and spreadsheet grid performance.
  * **Native Control Desync:** If users pause using the native context menu, the UI will desync unless you hook React state updates directly into the native `play` and `pause` event handlers.

### 3. Keyboard Left/Right Frame-Stepping
* **Cleanest Approach:**
  * **Keyboard Listener:** Register a `keydown` event listener in a `useEffect` hook. Add a `tabIndex={0}` to the review panel container to allow direct focus, or bind to `window` while the panel is active.
  * **Typing/Focus Guard:** Ensure keystrokes are ignored when users are typing comments:
    ```javascript
    const active = document.activeElement;
    const isTyping = active && (
      active.tagName === 'INPUT' || 
      active.tagName === 'TEXTAREA' || 
      active.hasAttribute('contenteditable')
    );
    if (isTyping) return;
    ```
* **Top Pitfalls:**
  * **Global Event Pollution:** Using `window.addEventListener('keydown')` without a mounting wrapper or active panel checks will trigger frame steps globally—even when the user is trying to navigate other parts of the UI or spreadsheet tabs with arrows.
  * **Focus Hijacking:** Relying purely on the player having focus (`tabIndex`) means frame-stepping breaks if the user clicks anywhere else in the version panel. A scoped global listener with an input guard is generally more robust.

### 4. Range Streaming from FastAPI / Starlette
* **Cleanest Approach:**
  * FastAPI's default `StaticFiles` middleware automatically handles HTTP Range requests out of the box, responding with a `206 Partial Content` status code and sending `Accept-Ranges: bytes` headers.
  * Verify the serving structure in your routes: `app.mount("/static", StaticFiles(directory="..."), name="static")`.
* **Top Pitfalls:**
  * **Middleware Interception:** Custom authentication or compression middlewares (like GZipMiddleware) can strip the `Range` headers or buffer response streams into memory, breaking range requests.
  * **Safari Playback Failures:** Apple Safari strictly refuses to decode or seek HTML5 videos if the server does not support byte ranges properly. Double check ranges using curl:
    ```bash
    curl -I -H "Range: bytes=0-100" http://localhost:8000/static/sample.mp4
    ```

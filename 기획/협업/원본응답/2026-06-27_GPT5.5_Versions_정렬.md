**Q1: Versions Grid Reuse**

- **Recommendation: keep `EntityGrid` as the reusable table shell, but split Shot/Asset-only features behind props.** Do not fork a full `VersionsGrid` unless review-specific behavior starts dominating the table. The current [EntityGrid.tsx](/Users/ghchoi/Workspace/active/study/FTPR_copy/개발/frontend/src/components/EntityGrid.tsx) already owns valuable generic behavior: fixed table layout, column widths, reorder, hide/show, grouping, selection, inline edit, thumbnails.

- **Make entity identity broader.**
  Current props are too narrow:
  ```ts
  entityKey: "shots" | "assets";
  entityType: "Shot" | "Asset";
  ```
  Change toward:
  ```ts
  entityKey: string;          // "shots" | "assets" | "versions"
  entityType: string;         // "Shot" | "Asset" | "Version"
  customFields?: false | {...}
  bulkStatus?: false | {...}
  ```
  Versions should pass `customFields={false}` and either `bulkStatus={false}` or a Version-specific status mutation if supported.

- **Best shape: feature props, not entity conditionals.**
  Avoid `if (entityType === "Version")` inside the grid. Prefer:
  ```ts
  customFields?: {
    defs: CustomFieldDef[];
    getValues(row): CustomValues;
    onCreate(...): void;
    onDelete(...): void;
    onUpdate(row, fieldId, value): void;
  }

  bulkActions?: ReactNode | false
  ```
  Shot/Asset pages can wrap backend custom-field hooks and pass them in. Versions pass nothing.

- **Use a small `VersionsGrid` wrapper only as composition.**
  A dedicated `VersionsGrid` can define Version columns and selection behavior, then render `EntityGrid<Version>`. That is cleaner than duplicating table mechanics:
  ```tsx
  <EntityGrid
    entityKey="versions"
    entityType="Version"
    columns={versionColumns}
    customFields={false}
    onSelect={setSelectedVersion}
  />
  ```

- **Top pitfalls:**
  - Current custom-field hooks are unconditionally called, so simply passing Version will still query/mutate custom fields unless gated.
  - `useUpdateStatus(entityKey)` currently expects `"shots" | "assets"`; bulk status must be optional or generalized before Versions use it.
  - `getCustomFields` is currently required; make it part of `customFields`, not a base grid prop.
  - LocalStorage view key must include `"versions"` so Shot/Asset column layout does not collide.
  - Row click for Versions should select/open review panel; code/entity link clicks must `stopPropagation()` to avoid double actions.

**Q2: Header Sorting**

- **Recommendation: add sorting to `GridColumn<T>` via an optional accessor.**
  ```ts
  sortAccessor?: (row: T) => string | number | boolean | Date | null | undefined;
  sortType?: "string" | "number" | "date" | "boolean";
  ```
  Keep render separate from sort. Never sort by rendered React content.

- **Sort state should be grid-local and column-key based.**
  ```ts
  type SortDir = "asc" | "desc";
  const [sort, setSort] =
    useState<{ key: string; dir: SortDir } | null>(null);
  ```
  Header click cycles:
  `none -> asc -> desc -> none`.

- **Grouping recommendation: filter -> group -> sort rows inside each group.**
  This preserves the user’s grouping mental model. If grouped by Status, groups stay Status groups, and rows inside each status group sort by Date/Artist/etc.
  
  Pipeline:
  ```ts
  rows
    -> search/filter
    -> group
    -> sort each group.rows
  ```
  Do not sort globally before grouping unless you specifically want group creation order to be affected by sort, which is usually surprising.

- **Group ordering: keep current group order first.**
  For now, do not sort group headers unless you add explicit `groupSortAccessor`. Default Map insertion order is acceptable for current behavior. Later:
  ```ts
  GroupOption<T> {
    accessor(...)
    sortAccessor?: (group) => string | number
  }
  ```

- **Header click vs drag/resize: make only the label/button sort.**
  Current header has a drag handle `.th-drag` and resize grip `.th-resize`. Do not put sorting on the whole `<th>`. Put it on `.th-label` or a nested button:
  ```tsx
  <button className="th-sort" onClick={() => toggleSort(c.key)}>
    {c.label}
    {active && <span>{dir === "asc" ? "▲" : "▼"}</span>}
  </button>
  ```
  Keep `onPointerDown/onClick` on resize grip stopped. Keep drag handle separate and stopped.

- **Only sortable columns should look clickable.**
  If `!c.sortAccessor`, render a plain label and no arrow. This avoids implying thumbnail/status widgets sort when no logic exists.

- **Active sort indicator: show arrow beside label and add ARIA.**
  Use:
  ```tsx
  aria-sort={active ? (dir === "asc" ? "ascending" : "descending") : "none"}
  ```
  Visually: muted arrow for inactive is optional; strongest signal is arrow only on active column.

- **Comparison pitfalls:**
  - Use stable sort by carrying original index:
    ```ts
    list.map((row, i) => ({ row, i })).sort(... || a.i - b.i)
    ```
  - Normalize nulls consistently, usually nulls last.
  - Use `Intl.Collator(undefined, { numeric: true, sensitivity: "base" })` for strings like `shot_2` vs `shot_10`.
  - Dates should compare timestamps, not locale strings.
  - Booleans should compare as `0/1`.
  - Do not mutate `filtered` or `rows`; always sort copies.

- **Version column accessors:**
  - Status: `v.status_name`
  - Version Name: `v.code`
  - Link/entity: `${v.entity_type}:${v.entity_name}`
  - Artist: `v.user?.name`
  - Task: task label/id if available
  - Notes count: `v.note_count`
  - Date: `Date.parse(v.created_at)`
  - Thumbnail: probably not sortable unless using `version_number` or omit sorting.

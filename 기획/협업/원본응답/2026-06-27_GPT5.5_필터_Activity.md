**A. Per-Column Filtering**

Recommendation: add a **global Filter toolbar popover/panel** first, powered by column metadata. It fits your current `EntityGrid` best because headers already carry drag, resize, and sort interactions; adding per-header popovers risks event conflicts and cramped fixed-layout headers.

Add to `GridColumn<T>`:

```ts
filter?: {
  type: "text" | "select" | "number";
  accessor: (row: T) => string | number | null | undefined;
  options?: { value: string; label: string; color?: string }[];
};
```

Clean flow:

- Keep `search` as global quick search.
- Add `filters: Record<string, FilterState>` in `EntityGrid`.
- Compute pipeline as: `rows -> search -> column filters -> group -> sort within groups`.
- Multiple filters should be **AND** by default, matching ShotGrid-style narrowing.
- Filter button opens a compact panel listing filterable visible/all columns: Status multi-select, text contains, number min/max.
- Persist filters in `localStorage` alongside grid view config, probably under `${entityKey}:filters`.

Why not filter row: it works for spreadsheets, but your table already has draggable/resizable/sortable headers and fixed widths. A filter row also becomes awkward for thumbnail mode.

Why not per-column popover first: good later, but more UI plumbing. It needs click targets in every header and careful resize/sort drag separation.

Top pitfalls:

- Use `filterAccessor`, not `render`, because rendered cells include React nodes/colors.
- Decide whether hidden columns can still filter. ShotGrid often allows this; simpler MVP is “all filterable columns,” with active filters shown as chips.
- Filter before grouping so group counts reflect visible rows.
- Sort only inside groups after filtering, as your code already does.
- For custom fields, generate filters from `CustomFieldDef.type`: `text`, `number`, `checkbox/select`.
- Memoize accessors and avoid rebuilding `colMap`-derived arrays carelessly; fine for hundreds/thousands of rows, not server-scale.

**B. Activity / History**

Recommendation: add an explicit `EventLog` model plus a small helper function, called from routers after meaningful mutations. Avoid SQLAlchemy listeners and middleware for this app.

Model shape is right:

```py
EventLog(
  id,
  project_id,
  entity_type,
  entity_id,
  event_type,
  attribute,
  old_value,
  new_value,
  user_id,
  created_at,
)
```

Use JSON/Text for old/new values. If values can be lists/custom fields, JSON is cleaner than string.

Helper approach:

- `record_event(db, project_id, entity_type, entity_id, event_type, attribute=None, old_value=None, new_value=None, user_id=None)`
- `record_field_changes(db, entity, entity_type, entity_id, project_id, before: dict, after: dict, fields: list[str])`
- Call it in `shots.py`, `assets.py`, `tasks.py`, `versions.py`, `notes.py`.
- Commit event logs in the same transaction as the domain change.

Why helper: your routers are simple and explicit. You need old/new values, entity link targets, and noise control. Listeners make old/new capture harder for JSON/custom fields and relationship changes; middleware cannot know semantic changes.

API shape:

- `GET /api/activity?project_id=1&limit=50`
- `GET /api/activity/entity/{entity_type}/{entity_id}`
- Optional filters later: `event_type`, `user_id`, `since`.

Frontend:

- Add `Activity` tab next to Tasks/Versions/Notes.
- Render newest-first or oldest-first consistently; I’d use newest-first for feeds, oldest-first only inside conversation-like notes.
- Event copy should be derived frontend-side or serializer-side:
  - `changed Status from Waiting to In Progress`
  - `created Note "Client feedback"`
  - `created Version v003`
- Add global Activity/Inbox page using the same API.

Top pitfalls:

- Capture `before` before mutating. In `update_shot`, snapshot fields before `setattr`, including `custom_fields`.
- JSON mutation tracking: assigning `s.custom_fields = merged` is good; in-place mutation may not persist or compare reliably.
- Avoid logging derived changes like `cut_duration` unless users edit it directly; otherwise it creates noise.
- Do not log every unchanged PATCH field. Compare normalized old/new values.
- For no-auth prototype, use `user_id=None` or a fixed demo user/header like `X-User-Id`; choose one and keep it consistent.
- Add indexes: `(entity_type, entity_id, created_at)` and `(project_id, created_at)`.
- For notes/versions linked to Shot/Asset, record activity both on the created entity itself and optionally on the linked parent. For MVP, record on the linked parent so the entity detail Activity tab is useful.

### Feature A: Checkbox Bulk Selection

#### Recommended Approach
* **Selection State & Integration**: Manage selection as a `Set<string>` of row IDs in the grid's parent state. Render a hardcoded checkbox column at index 0 of the `colgroup` and `thead`/`tbody`, explicitly bypassing the user-configurable `columns` array. This prevents the column from being reordered by drag-and-drop, hidden via the Fields menu, or resized.
* **Select-All Logic**: The header checkbox should display an `indeterminate` state (managed via a React `ref`) when some, but not all, visible rows are selected. Clicking "Select All" must only add or remove currently filtered/visible row IDs to/from the selection set, rather than invisible or filtered-out rows.
* **Event Separation**: Prevent event bubbling on the checkbox wrapper by using `e.stopPropagation()` in the checkbox's `onChange` or `onClick` handler. This keeps the checkbox selection separate from row selection (`onClick`) and inline editing (`onDoubleClick`).
* **Bulk-Action Overlay**: Render the bulk-action bar as a fixed or absolute overlay centered at the bottom of the viewport, which avoids shifting the layout of the `<table>` when selection toggles. Perform the status bulk-update by sending a batch patch request (or executing concurrent `PATCH` calls), then trigger a grid data refresh.

#### Top 3 Pitfalls
1. **Unintentional Selection of Filtered Rows**: If "Select All" simply selects all records in memory, rows hidden by search queries or active filters will be modified during a bulk status change. 
2. **Double-Click Edit Conflict & Event Bubbling**: Clicking the checkbox can trigger parent row-click events, causing row selection styles to flash or launching the inline cell editor.
3. **Sticky Column Overflow & Layering**: If the checkbox column is sticky (`left: 0`), cell contents from scrolling columns will bleed over it unless it is explicitly styled with a background color and a higher `z-index` (e.g., `z-index: 2`).

### Feature B: Schedule / Gantt View

#### Recommended Approach
* **Layout & Rendering**: Use a hybrid CSS Grid and absolute positioning layout. The timeline header and grid lines are represented by a CSS Grid (e.g., one column per day or week tick). Inside each row, render a single relative-positioned container where task bars are positioned absolutely using percentage-based `left` and `width` values calculated from the timeline's date boundaries.
* **Date-to-Pixel Mapping**: Establish a viewport date range (`viewStart` to `viewEnd`). Calculate the total duration in days. For each task, compute percentage positions:
  * `left = (daysBetween(task.start_date, viewStart) / totalDays) * 100`
  * `width = (daysBetween(task.due_date, task.start_date) / totalDays) * 100`
* **Data Structure**: Return a flat list of entities containing nested, pre-sorted task arrays from the backend:
  ```json
  {
    "entities": [
      {
        "id": "asset_101",
        "name": "Hero_Asset",
        "tasks": [
          { "id": 1, "step": "Model", "status": "IP", "start_date": "2026-07-01", "due_date": "2026-07-10" }
        ]
      }
    ]
  }
  ```
* **Zoom Levels**: Support a state-driven `zoom` ('day', 'week', 'month') that dynamically updates the day-width coefficient and changes the grid's column header interval ticks.

#### Top 3 Pitfalls
1. **Daylight Saving Time & Timezone Shifts**: Doing raw millisecond math (`86400000` ms per day) to calculate date offsets will result in +/- 1 day offsets during DST transitions. Use UTC-based or start-of-day normalized date-fns/Luxon operations.
2. **Layout Thrashing with Large Datasets**: Rendering hundreds of rows with dozens of tasks across a wide timeline generates thousands of DOM nodes, causing severe scroll lag. Implement row-based vertical virtualization.
3. **Missing or Incomplete Dates**: Tasks with null `start_date` or `due_date` values will evaluate to `NaN` and crash the CSS styling. Provide a fallback visual state (e.g., rendering a warning badge or a dashed "unscheduled" block at the current date).

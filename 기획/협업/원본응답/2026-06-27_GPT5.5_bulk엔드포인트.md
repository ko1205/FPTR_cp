1. **REST Shape**
   - Use explicit sub-routes:
     - `POST /api/shots/batch`
     - `POST /api/shots/batch-delete`
   - Avoid `DELETE` with JSON body; it is legal but less consistently supported by clients, proxies, and OpenAPI tooling.
   - Mirror per-entity routes: `/api/assets/batch`, `/api/tasks/batch-delete`, etc.

2. **Transaction Semantics**
   - For this prototype: use **all-or-nothing** in one transaction.
   - Validate all rows first, create/delete all, write EventLogs, then commit once.
   - Cleaner frontend behavior, simpler error handling, and no confusing partial EventLog history.
   - Best-effort can come later if CSV/import workflows need row-level recovery.

3. **Activity Logging**
   - Prefer **one EventLog per affected entity**.
   - ShotGrid-style activity is entity-centric: opening a Shot/Asset/Task should show that it was created/deleted/updated.
   - For very large batches, you can additionally add one optional summary event, but do not replace per-entity history with only a summary.

4. **Response Shape**
   - Bulk create:
     ```json
     {
       "created": [/* full serialized entities */],
       "count": 12
     }
     ```
   - Bulk delete:
     ```json
     {
       "deleted_ids": [1, 2, 3],
       "count": 3
     }
     ```
   - On validation failure, return `422` with row/index-specific errors.
   - Frontend can simply invalidate the entity list, activity feed, and related project stats queries once after success.

- **I mostly agree with your plan**, with one adjustment: keep **view config** in `localStorage`, but consider storing **custom field definitions + values** in SQLite if the prototype must feel like real ShotGrid across refreshes, browsers, users, or seeded demos.

- **1. Custom field values**
  - `localStorage` is acceptable for a solo FE prototype and matches the current `useGridView` / `useCustomValues` shape.
  - For a ShotGrid-like feel, I’d prefer SQLite/API:
    - Add custom field definitions per entity type.
    - Store values as JSON keyed by field id, either in a `custom_fields` JSON column or a side table.
  - Better split:
    - `localStorage`: column order, hidden fields, thumbnail/list mode, maybe saved UI prefs.
    - SQLite/API: custom field schema + row values.
  - Reason: ShotGrid custom fields are part of the production schema, not just a browser preference.

- **2. Native HTML5 DnD vs `dnd-kit`**
  - Native is enough for a desktop-only prototype if you keep it simple.
  - Pitfalls with `<table>` headers:
    - Awkward drag previews.
    - Poor touch support.
    - Weak keyboard/accessibility story.
    - Browser quirks around `draggable` on table elements.
    - Harder to show precise before/after drop indicators.
    - Can conflict with header controls later: sort, resize, field menu.
  - If this grid is going to grow, use `dnd-kit`.
  - If this is just a prototype demo, native is fine.

- **3. ShotGrid behaviors you’re missing**
  - Column resizing.
  - Frozen/pinned columns, especially Status / Thumbnail / Code.
  - Saved views per entity/project/user.
  - Per-column sort.
  - Per-column filters, not just global search.
  - Inline editing for normal fields, not only status/custom cells.
  - Field types beyond text/number/checkbox:
    - date
    - status/list
    - entity link
    - multi-entity link
    - user/person
    - duration/frame fields
    - URL
  - Required/default values for custom fields.
  - Rename/delete custom fields with value cleanup rules.
  - Bulk edit selected rows.
  - Column width persistence.
  - Row density / thumbnail size controls.

- **Recommended direction**
  - Keep your current plan for fast prototype velocity.
  - Upgrade custom fields to SQLite/API if “feels like ShotGrid” means shared, persistent, schema-like behavior.
  - Use native DnD now, but avoid coupling the reorder logic too tightly to native events so it can be swapped for `dnd-kit` later.

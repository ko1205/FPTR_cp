Given this codebase, keep `EntityGrid` generic but let pages pass CRUD capabilities/config into it. `EntityGrid` should own UI mechanics: menu positioning, selected ids, modal state. Pages should own entity-specific create/delete mutation wiring, field config, navigation, and payload shaping.

**A) Row Right-Click Menu**

- Add `onContextMenu` to the row in `GroupRows`.
- Call `e.preventDefault()` to suppress the native browser menu.
- Call `e.stopPropagation()` so right-click does not behave like row selection.
- Store menu state in `EntityGrid`:
  ```ts
  const [ctxMenu, setCtxMenu] = useState<{ x: number; y: number; row: T } | null>(null);
  ```
- Use `e.clientX / e.clientY` for fixed-position menu coordinates:
  ```tsx
  <div className="context-menu" style={{ position: "fixed", left: x, top: y }} />
  ```
- Close on:
  - document `pointerdown` outside
  - `Escape`
  - scroll/wheel
  - route/open action
- Do not attach context menu to the checkbox cell. That cell already uses `stopPropagation`.
- Confirm delete with a modal, not `window.confirm`, if you want a ShotGrid-like feel:
  - title: `Delete Shot SH010?`
  - body: explain it cannot be undone
  - destructive red `Delete`
  - neutral `Cancel`
- Recommendation: implement context menu UI inside `EntityGrid`, but expose actions via props:
  ```ts
  rowActions?: Array<{
    key: string;
    label: string;
    danger?: boolean;
    onRun: (row: T) => void;
  }>
  ```
- Pitfalls:
  - Right-click accidentally selecting the row.
  - Menu going off-screen near bottom/right; clamp to viewport after render or use estimated width/height.
  - Forgetting to close on table scroll.
  - Opening delete while an inline edit is active.
  - Deleting a row but leaving `selectedKey` or bulk `selectedRows` pointing at a removed id.

**B) Bulk Delete**

- Use the existing `selectedRows: Set<number>` in `EntityGrid`.
- Enable the disabled `More` button and show `Delete selected (N)` when `selCount > 0`.
- Require confirmation:
  - `Delete 12 selected shots?`
  - Show count, not every item, unless count is small.
- For now, sequential `DELETE /shots/{id}` is acceptable:
  ```ts
  await Promise.allSettled([...selectedRows].map(id => deleteEntity.mutateAsync(id)));
  ```
- Prefer `Promise.allSettled`, not plain `Promise.all`, so one failure does not hide the rest.
- After completion:
  - invalidate/refetch entity query
  - clear successfully deleted ids from `selectedRows`
  - keep/report failed ids
- Recommendation:
  - Short term: loop DELETE requests from frontend using `mutateAsync`.
  - Long term: add backend endpoint `POST /shots/bulk-delete` or `DELETE /shots` with `{ ids }`.
- Pitfalls:
  - Sequential one-by-one deletes are slow for 100+ rows.
  - Parallel deletes can overload local SQLite/FastAPI if unbounded; use small concurrency if needed.
  - Clearing all selection even when partial failures occurred.
  - Optimistically removing rows without rollback handling.
  - Cascading deletes: versions/tasks/notes may make delete semantically dangerous. Backend should own cascade rules.

**C) Generic Add Form**

- Do not bake Shot/Asset fields into `EntityGrid`.
- Add a generic create config prop:
  ```ts
  type CreateField = {
    key: string;
    label: string;
    type: "text" | "number" | "textarea" | "select";
    options?: Array<{ label: string; value: string | number }>;
    required?: boolean;
    default?: unknown;
  };

  createConfig?: {
    fields: CreateField[];
    primaryLabel?: string;
    onCreate: (data: Record<string, unknown>) => Promise<unknown>;
  };
  ```
- `Shots.tsx` supplies fields like:
  - `code`
  - `sequence_id`
  - `description`
  - `cut_in`
  - `cut_out`
  - `status`
  - `project_id` as hidden/default payload
- `EntityGrid` renders the modal from config and calls `onCreate(payload)`.
- Keep payload shaping in the page if entity-specific:
  ```ts
  onCreate={(data) => createShot.mutateAsync({ ...data, project_id: projectId })}
  ```
- Recommendation: use a modal create form, not inline new-row.
- Why modal:
  - easier validation
  - easier select fields
  - no conflict with sorting/grouping/filtering
  - no fake row lifecycle
  - works for all entities
- Pitfalls:
  - Number inputs still return strings; coerce before POST.
  - Select options may depend on loaded reference data like sequences/statuses.
  - Required validation should happen before POST, but backend must still validate.
  - Defaults must update when `projectId` changes.
  - After creation, decide whether to clear filters/search or leave the new row possibly hidden.

**D) Add Multiple / Add CSV**

- Add these under `More`, backed by the same `createConfig`.
- `Add Multiple`:
  - textarea
  - one line per entity
  - each line maps to the primary field, usually `code`
  - apply shared defaults: `project_id`, default status, etc.
- Example payloads:
  ```ts
  lines.map(code => ({ project_id: projectId, code, status: "wtg" }))
  ```
- `Add CSV`:
  - paste CSV with header row
  - parse with a real parser, not `split(",")`
  - use `papaparse` in frontend or Python/csv backend-side
  - map headers to create fields
  - preview parsed rows before submit
- Recommendation:
  - For MVP: frontend parse + loop `POST`, using `Promise.allSettled`.
  - For production: backend bulk endpoint `POST /shots/bulk` returning per-row success/error results.
- Pitfalls:
  - CSV quoting: commas inside quoted fields, escaped quotes, newlines inside quoted cells.
  - Header mismatches: `Shot Code`, `code`, `shot_code` need mapping/normalization.
  - Empty lines and duplicate codes.
  - Required fields missing from CSV.
  - Bad number parsing.
  - Select labels vs select values, e.g. `Waiting` vs `wtg`.
  - Partial failures: user needs row-level errors and should not lose the original pasted data.
  - Bulk creates should not invalidate/refetch after every row; batch invalidation once at the end.

Cleanest overall shape:

```ts
<EntityGrid
  ...
  createConfig={shotCreateConfig}
  rowActions={shotRowActions}
  bulkActions={shotBulkActions}
/>
```

Let `EntityGrid` provide generic UI surfaces, while `Shots.tsx`, `Assets.tsx`, etc. provide entity-specific fields, mutations, labels, and navigation. That keeps the grid reusable without turning it into a backend-aware CRUD god component.

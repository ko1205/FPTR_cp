- **A) Checkbox Bulk Selection**
  - **Recommended approach**
    - Add bulk selection as first-class `EntityGrid` state: `selectedIds: Set<number>`, `toggleRow(id)`, `clearSelection()`, `selectVisible()`.
    - Add a fixed internal selection column before `visibleColumns`, not inside `columns`, `cfg.order`, or `FieldsMenu`.
    - Header checkbox should use `filtered` visible rows, preferably excluding collapsed group rows only if your mental model is “currently rendered rows”; otherwise use all filtered rows across groups.
    - Row checkbox `onClick/onChange` must `stopPropagation()` so it does not trigger `onSelect`.
    - Keep existing single-row `selectedKey` for detail focus, and use a separate class like `row-bulk-selected` for bulk highlight.
    - Bulk status action should live in `EntityGrid` as a render prop, e.g. `renderBulkActions({ selectedIds, selectedRows, clear })`, while `Shots` supplies the status mutation behavior.
    - For PATCH: either sequential with `Promise.allSettled`, or better add backend bulk endpoint later: `PATCH /shots/bulk { ids, data }`.

  - **Top 3 pitfalls**
    - **Select-all semantics:** decide explicitly whether select-all means all `filtered` rows, all currently expanded/rendered rows, or all rows matching server-side filters. For ShotGrid-like UX, use “currently visible/filtered rows” and show `N selected`.
    - **Event conflicts:** checkbox clicks, status dropdowns, custom checkbox fields, inline edit inputs, links, and thumbnails must all stop propagation so row selection/open/edit behavior stays predictable.
    - **Selection column integration:** do not let the checkbox column be reorderable, resizable, hideable, or part of custom field config. Give it a constant width like `36px`; optionally make it sticky only if horizontal scrolling is already painful. Sticky first column is useful, but adds z-index/background edge cases with group headers and table layout.

- **B) Schedule / Gantt View**
  - **Recommended approach**
    - Build a separate `ScheduleView` page/component, not inside `EntityGrid`.
    - Use a two-pane layout: left fixed row labels, right horizontally scrollable timeline.
    - Use CSS grid/absolute positioning for bars:
      - timeline container: `position: relative`
      - each row: fixed height, e.g. `32px`
      - each task bar: `position: absolute; left: dateToX(start); width: dateToX(due + 1) - dateToX(start)`
    - Use pixels-per-day as the core scale:
      - day zoom: `24px/day`
      - week zoom: `8-12px/day`
      - month zoom: `2-4px/day`
    - Compute visible range from task min/max dates with padding, e.g. 7-14 days.
    - Render ticks from the same date scale: month bands + week tick lines is a good first version.
    - Backend should return denormalized task rows:
      ```ts
      {
        id,
        code,
        entity_type,
        entity_id,
        entity_code,
        step_id,
        step_name,
        status_code,
        status_name,
        status_color,
        start_date,
        due_date
      }
      ```
    - Keep grouping client-side at first: by `entity_code`, `step_name`, or status.

  - **Top 3 pitfalls**
    - **Date math/timezone:** treat `start_date` and `due_date` as date-only values, not JS local datetimes. Parse as UTC or manual `YYYY-MM-DD` parts to avoid one-day shifts.
    - **Missing/sparse dates:** define behavior early. Hide unscheduled tasks in an “Unscheduled” group, or render them as label-only rows. Do not let null dates break range calculations.
    - **Performance and future dragging:** absolute-positioned DOM bars are fine for hundreds of tasks. For thousands, add row virtualization. If drag-to-reschedule is planned, keep all positioning derived from `dateToX/xToDate` helpers so snapping, zooming, and PATCH updates reuse the same model.

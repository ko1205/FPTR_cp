### 1. Layout & Scroll (Horizontal & Vertical Scrolling Table)

*   **Recommendation**:
    *   **Wrapper CSS**: Wrap the table in a container `div` styled with `overflow: auto`, `position: relative`, and `height: 100%; width: 100%;`.
    *   **Table CSS**: Apply `table-layout: fixed` and `width: max-content` (or style the width as the sum of all columns) to the `<table>` element. This prevents the table from shrinking below its explicit column widths.
    *   **Sticky Headers**: To keep headers visible during vertical scrolling, style `thead th` with `position: sticky; top: 0; z-index: 2;` along with a solid background color. Use `position: sticky; left: 0; z-index: 3;` for frozen columns (like Shot/Asset Name).
*   **Pitfalls**:
    *   Using `width: 100%` on the table causes browser layout engines to shrink columns when the container is smaller than the combined column widths. You must force the table layout sizing using `width: max-content` or explicit pixel measurements.
    *   `position: sticky` on the header will fail if any parent container between the wrapper and the table has `overflow: hidden` or `overflow: scroll` set.

### 2. Column Resize

*   **Recommendation**:
    *   **HTML Structure**: Put a `<colgroup>` at the very top of the table containing `<col style={{ width: columnWidths[colId] }} />` elements. This updates column sizes instantaneously without triggering expensive layout reflows across every individual row cell.
    *   **Resize Grip**: Inside each `<th>`, render a thin absolute-positioned target on the right border.
    *   **Pointer Capture**: Bind `onPointerDown` to the resize grip. Call `e.currentTarget.setPointerCapture(e.pointerId)` to capture mouse movement globally, then handle resizing using standard `onPointerMove` and `onPointerUp` listeners on the grip element itself. Debounce or throttle updates to `localStorage` on pointer release.
*   **Pitfalls vs. TanStack Table**:
    *   *Writing from scratch*: You must manually calculate drag deltas, enforce minimum/maximum boundaries (e.g., minimum 80px), prevent browser text selection during drags, and handle high-DPI zoom offsets.
    *   *TanStack Table*: Resolves these edge cases but introduces unnecessary architectural complexity and boilerplate if you only need basic drag-resize logic on top of a native HTML table.

### 3. Double-Click Edit

*   **Recommendation**:
    *   **State Management**: Maintain an `editingCell` state mapping to `{ rowId, columnKey }`.
    *   **Conditional Render**: When `editingCell` matches the current cell, replace the read-only cell contents with a text `<input>` or `<textarea>`.
    *   **Focus & Save**: Auto-focus the input using a `ref` or `autoFocus` property when it mounts. On `onBlur` or `onKeyDown` (detecting `Enter`), save changes using the PATCH API callback and clear the `editingCell` state. Allow `Escape` to discard edits.
*   **Pitfalls & Gotchas**:
    *   **Event Propagation**: Double-clicks will bubble up and trigger row selection. Use `e.stopPropagation()` on both double-clicks and edits to prevent row selection state toggling or triggering drag handlers.
    *   **Race Conditions**: Both `onBlur` and pressing `Enter` can fire simultaneously. Ensure your save handler ignores duplicate calls to avoid firing redundant PATCH requests.
    *   **Drag-to-Reorder Interference**: If headers or rows are draggable, make sure `draggable="true"` is only active when not editing, and that drag handles are isolated from cell text selection areas.

### 4. Pipeline Column (Step Alignment)

*   **Recommendation**:
    *   **Align by Step**: Aligning by step is the correct and faithful ShotGrid behavior. Columns should represent project-wide pipeline stages (e.g., Layout, Anim, FX, Comp) in a fixed layout sequence.
    *   **Implementation**: 
        1. Query the project schema for a ordered master list of pipeline steps: `['Layout', 'Animation', 'FX', 'Lighting', 'Compositing']`.
        2. Render a parent `<th colspan={steps.length}>Pipeline</th>` with sub-headers representing each step.
        3. Inside `<EntityGrid>` rows, map over this master step array. If the row asset contains a task matching the step ID, render the status cell. If not, render an empty placeholder cell (`<td>` containing a light gray dash or spacer) to preserve step alignment.
*   **Pitfalls**:
    *   A variable-length strip of task cells destroys vertical scannability. If "Lighting" is the first task for Asset A but the fourth task for Asset B, the user cannot scan down a single visual column to assess the status of "Lighting" across all assets.
    *   Master lists of steps must be computed and ordered consistently. Inconsistent schema ordering will cause the sub-columns to shift layout when sorting or filtering table rows.

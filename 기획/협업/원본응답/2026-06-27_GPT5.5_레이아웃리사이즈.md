- **1) Layout/Scroll**
  - Recommendation: put the grid in a dedicated scroll wrapper and make the table width content-driven, not viewport-driven.
  - CSS shape:
    ```css
    .gridViewport {
      width: 100%;
      height: calc(100vh - var(--top-bars-height));
      overflow: auto;
    }

    .entityTable {
      table-layout: fixed;
      border-collapse: collapse;
      width: max-content;
      min-width: 100%;
    }

    .entityTable th,
    .entityTable td {
      overflow: hidden;
      text-overflow: ellipsis;
      white-space: nowrap;
    }
    ```
  - Apply explicit widths through `<colgroup>`:
    ```tsx
    <table className="entityTable">
      <colgroup>
        {columns.map(col => (
          <col key={col.key} style={{ width: columnWidths[col.key] }} />
        ))}
      </colgroup>
    </table>
    ```
  - Use pixel widths for grid columns, not `%` or flex sizing.
  - Pitfall: `width: 100%` on the table will cause columns to compress unless total column width exceeds the container and the table is allowed to grow. Prefer `width: max-content; min-width: 100%`.
  - Pitfall: if cells contain images or buttons, give them stable dimensions so they do not expand the column unexpectedly.
  - Pitfall: sticky headers/columns require the scroll container to be the same element that owns `overflow: auto`.

- **2) Column Resize**
  - Recommendation: implement resize yourself unless you are already adopting TanStack Table for broader table state management.
  - Use a small resize grip inside each `<th>`, with pointer events:
    ```tsx
    <div
      className="resizeGrip"
      onPointerDown={e => startResize(e, column.key)}
    />
    ```
  - Store widths as:
    ```ts
    Record<string, number>
    ```
    then apply them to `<colgroup><col style={{ width }} />`.
  - During drag, update width from `startWidth + event.clientX - startX`.
  - Use `document`-level `pointermove` / `pointerup`, or pointer capture, so resizing continues even if the cursor leaves the header.
  - Persist to localStorage by grid/entity type:
    ```ts
    entityGrid:shots:columnWidths
    entityGrid:assets:columnWidths
    ```
  - Add `minWidth` per column, especially thumbnail/status/action columns.
  - Pitfall: resizing and header drag-to-reorder will conflict. The resize grip must call `stopPropagation()` and header drag should ignore events started from the grip.
  - Pitfall: do not update every cell’s inline width. Set width once in `<colgroup>`.
  - Pitfall: TanStack Table is excellent if you need sorting/filtering/virtualization/column pinning later, but migrating an existing custom grid may be more churn than value if your current table already works.

- **3) Double-Click Edit**
  - Recommendation: keep this cell-scoped. Double-click enters edit mode for that cell only; blur or Enter saves; Escape cancels.
  - Prevent row selection from firing while editing:
    ```tsx
    onDoubleClick={e => {
      e.stopPropagation();
      setEditingCell({ rowId, columnKey });
    }}
    ```
  - The input should also stop pointer/click/key propagation where needed.
  - Save behavior:
    - `Enter`: commit PATCH
    - `Blur`: commit PATCH if dirty
    - `Escape`: revert local value
  - Use optimistic UI only if failed saves are visibly handled. Otherwise keep a pending state on the cell.
  - Pitfall: blur and Enter can double-save. Centralize commit logic and guard with `isSaving` / `hasCommitted`.
  - Pitfall: double-click may also trigger row click selection first. Either delay row selection slightly, or make row selection happen only from non-editable cell areas.
  - Pitfall: header drag is mostly separate, but if the grid uses document-level pointer handlers, ensure editing inputs are excluded from drag/select logic.
  - Pitfall: avoid replacing the whole row object during editing if that resets focus. Keep draft state local to the cell editor.

- **4) Pipeline Column**
  - Recommendation: align by pipeline step. That is more faithful to ShotGrid and much easier to scan across rows.
  - Variable-length strips are compact, but they lose positional meaning. A green third block may mean FX on one row and Lighting on another, which is bad for production tracking.
  - Define a canonical step order per entity type or project:
    ```ts
    const shotPipelineSteps = ['Layout', 'Anim', 'FX', 'Lighting', 'Comp'];
    ```
  - Normalize tasks per row:
    ```ts
    const taskByStep = new Map(shot.tasks.map(task => [task.stepCode, task]));
    ```
  - Render fixed sub-columns:
    ```tsx
    <div className="pipelineCells">
      {pipelineSteps.map(step => {
        const task = taskByStep.get(step);
        return (
          <div
            key={step}
            className="pipelineCell"
            data-empty={!task}
          >
            {task ? renderTaskStatus(task) : null}
          </div>
        );
      })}
    </div>
    ```
  - CSS:
    ```css
    .pipelineCells {
      display: grid;
      grid-template-columns: repeat(var(--step-count), 28px);
    }

    .pipelineCell {
      width: 28px;
      height: 22px;
    }
    ```
  - Store the whole pipeline column width as `steps.length * stepWidth`, or expose each step as a real sub-column if you want proper sorting/filtering per step later.
  - Pitfall: different projects may have different pipeline templates. Do not hardcode globally unless your app only supports one pipeline.
  - Pitfall: task names and step names are not always the same. Align by stable step code/id, not display label.
  - Pitfall: missing task should render as an empty slot, not disappear.
  - Pitfall: if users can hide/reorder columns, decide whether pipeline steps are one compound column or individual grid columns early. Individual columns are more ShotGrid-like; one compound column is simpler.

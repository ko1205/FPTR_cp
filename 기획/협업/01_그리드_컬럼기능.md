## 1. 질문 (메인 → 두 모델 동일)

> React+TS+Vite ShotGrid 클론. 제네릭 `<EntityGrid>`(columns 배열로 구성)에
> (1) 헤더 드래그로 **컬럼 순서변경**, (2) **Fields 드롭다운으로 표시/숨김**,
> (3) **커스텀필드 추가**(text/checkbox/number) 를 넣어야 한다.
>
> **내 제안**: 엔티티별 뷰설정 `{order, hidden, customFields}` 를 localStorage 에 저장.
> 재배치는 네이티브 HTML5 draggable. 커스텀필드 **값은 SQLite 스키마 고정 제약 때문에
> 클라이언트 localStorage(엔티티 id 키)에 저장**.
>
> Q1. 커스텀필드 값: localStorage vs SQLite JSON 컬럼+API?
> Q2. 네이티브 HTML5 DnD vs dnd-kit?
> Q3. 빠진 ShotGrid 동작은?

(전문: [`원본응답/2026-06-27_질문_grid기능.txt`](원본응답/2026-06-27_질문_grid기능.txt))

## 2. 모델별 답변 요약

### 🟦 Gemini (3.5 Flash) — [전문](원본응답/2026-06-27_Gemini_grid기능.md)
- **Q1**: **SQLite JSON 컬럼 + API 권장.** ShotGrid은 본질적으로 협업 도구 →
  localStorage 면 다른 사용자에게 안 보여 협업 흐름이 깨짐. SQLite JSON 지원 견고,
  `PATCH /api/shots/{id}` 로 마이그레이션 없이 병합 가능. **뷰설정(순서/표시)만 localStorage.**
- **Q2**: **`@dnd-kit` 권장.** `<th>` 네이티브 DnD 는 레이아웃 붕괴·드롭 인디케이터 오프셋·
  애니메이션 부재·접근성/터치 약함.
- **Q3**: 컬럼 리사이즈, 파이프라인 스텝 컬럼(공정별 상태 묶음), 다중 엔티티 링크 필드,
  요약(summary) 행, 다중행 일괄편집.

### 🟩 GPT-5.5 (Codex, 내 코드 직접 읽음) — [전문](원본응답/2026-06-27_GPT5.5_grid기능.md)
- **Q1**: **대체로 동의하나**, "ShotGrid답게(공유·영속·스키마)" 하려면 **커스텀필드 스키마+값을
  SQLite/API 로**. 분담: localStorage=컬럼순서/숨김/뷰모드, SQLite=커스텀필드 스키마+값.
- **Q2**: 프로토타입엔 **네이티브로 충분**. 단 재배치 로직을 네이티브 이벤트에 **강결합하지
  말고** 나중에 dnd-kit 으로 교체 가능하게. (table 헤더 DnD 의 단점들 동일 지적)
- **Q3**: 컬럼 리사이즈, **고정(pinned) 컬럼**, 저장된 뷰, 컬럼별 정렬/필터,
  더 많은 필드 타입(date/list/entity/user/duration/URL), 일괄편집, 컬럼 너비 영속화.

## 3. 합의 / 이견 분석

| 쟁점 | Gemini | GPT-5.5 | 합의? |
|------|--------|---------|:----:|
| 뷰설정(순서/숨김) 저장 위치 | localStorage | localStorage | ✅ 합의(내 안과 동일) |
| **커스텀필드 값 저장** | **SQLite JSON+API** | **SQLite/API 권장** | ✅ **둘 다 내 localStorage 안에 반대** |
| 재배치 구현 | dnd-kit | 네이티브 OK(분리) | ⚠ 이견 |
| 공통 누락 지적 | 컬럼 리사이즈 등 | 컬럼 리사이즈 등 | ✅ 둘 다 리사이즈 강조 |

> **핵심**: 두 모델이 **독립적으로** "커스텀필드 값은 localStorage가 아니라 DB에" 라는
> 같은 결론에 도달. 이건 강한 신호.

## 4. 메인 에이전트(Claude)의 판단과 처리

### 채택
1. **뷰설정(order/hidden)은 localStorage 유지** — 만장일치, 내 구현 그대로.
2. **커스텀필드 정의+값을 SQLite 로 이전** — 두 모델 합의에 따라 변경.
   - 백엔드: Shot/Asset 에 `custom_fields` JSON(TEXT) 컬럼 추가, 커스텀필드 정의는
     `custom_field_defs` 테이블(project_id, entity_type, field_id, label, type).
   - API: `GET/POST/DELETE /api/custom_fields`, 값은 엔티티 PATCH 의 `custom_fields` 병합.
   - 프론트: 정의/값을 API 로 읽고 쓰도록 전환(뷰설정만 localStorage).
3. **재배치는 네이티브 유지**(GPT-5.5 의견 채택 — 프로토타입엔 충분),
   단 재배치 로직을 `moveColumn(from,to)` 로 분리해 추후 dnd-kit 교체 용이하게 함.

### 보류(다음 단계로 기록)
- 컬럼 **리사이즈**(둘 다 강조) → 우선순위 높은 다음 작업.
- 고정 컬럼, 컬럼별 정렬/필터, 요약 행, 일괄편집, 추가 필드 타입(date/entity/user 등).

### 비고
- 1차 구현은 내 원안(localStorage)대로 빠르게 완성 → 협업 결론 반영해 **2차로 SQLite 전환**.
  덕분에 동작은 즉시 검증하면서, 더 "원본다운" 영속/협업 모델로 개선.

### 구현·검증 결과 (완료)
- 백엔드: Shot/Asset 에 `custom_fields` JSON 컬럼 + `custom_field_defs` 테이블 +
  `/api/custom_fields` (GET/POST/DELETE) + 엔티티 PATCH 의 custom_fields 병합. ✅
- 프론트: 뷰설정(순서/숨김)=localStorage, 커스텀필드 정의/값=API. 헤더 드래그 재배치,
  Fields 메뉴(표시/숨김 + 커스텀필드 추가/삭제), 셀 인라인 편집(blur 저장). ✅
- **end-to-end 검증**: API로 만든 "Priority" 정의가 그리드에 컬럼으로 나타나고,
  PATCH한 값("High")이 DB에서 로드되어 셀에 표시됨.
  (스크린샷: `기획/이미지/원본참고/내클론_커스텀필드_Priority.png`)
- 빌드 `npm run build` 0 에러.

# 01. 연구 조사 — Flow Production Tracking 원본 제품 분석

본 문서는 프로토타입 설계의 근거가 된 **원본 FPTR(구 ShotGrid/Shotgun)** 의
도메인·기능 조사 내용을 정리한다.

---

## 1. 제품 정체성

FPTR은 **VFX / 애니메이션 / 게임 제작 스튜디오를 위한 프로덕션 관리(Production
Management) 웹 플랫폼**이다. 한 편의 영화/시리즈/게임을 만들기 위해 수백~수만 개의
샷(Shot)과 에셋(Asset)을 수십~수백 명의 아티스트가 동시에 작업하는데, 이 거대한
작업을 **추적(tracking)** 하고 **검토(review)** 하며 **협업(collaboration)** 하게 해준다.

### 핵심 가치 제안 3가지
1. **Track** — 무엇을, 누가, 어디까지 했는가 (Shot/Asset/Task 상태 추적)
2. **Review** — 만들어진 결과물(미디어)을 보고 피드백 (Version + Note)
3. **Manage** — 일정·할당·진척도 관리 (스케줄, 부하, 번다운)

> 경쟁/유사 제품: ftrack, Kitsu(오픈소스), Autodesk Flow(통합), Perforce/Helix.

---

## 2. 핵심 도메인 개념 (Entity 중심)

FPTR의 본질은 **유연한 엔티티-필드 데이터베이스 + 미디어 리뷰**다.
아래는 실제 FPTR의 표준 엔티티(공식적으로 "Entity Type"이라 부름)다.

### 2.1 조직/사람
- **HumanUser** — 사람 사용자 (아티스트, 슈퍼바이저, 코디네이터)
- **Group / Department** — 부서(예: Animation, Lighting, FX, Comp)
- **Project** — 최상위 컨테이너. 모든 데이터는 특정 프로젝트에 속함.

### 2.2 작업 대상 (제작물)
- **Asset** — 재사용 가능한 제작물. 캐릭터, 프롭, 환경, 차량 등.
  - `sg_asset_type`: Character / Prop / Environment / Vehicle / FX ...
- **Sequence** — 샷의 묶음(시퀀스). 영화의 한 장면 단위.
- **Shot** — 최소 연출 단위(연속된 카메라 컷 하나).
  - 속성: cut_in, cut_out, cut_duration(프레임), 썸네일 등
- **Scene** (선택) — 시퀀스/샷의 상위 또는 평행 그룹

### 2.3 작업(Work)
- **Task** — 실제 할 일. 항상 어떤 엔티티(Shot/Asset)에 붙는다.
  - 속성: 담당자(assignees), 상태(status), 시작/종료일, Pipeline Step
- **Pipeline Step (Step)** — 작업 공정 단계. 예: Modeling, Rigging, Animation,
  Lighting, FX, Comp. Task는 반드시 하나의 Step에 속한다.

### 2.4 리뷰(Review)
- **Version** — 아티스트가 제출한 결과물(미디어) 1개. "퍼블리시된 산출물의 검토용 표현".
  - 썸네일, 동영상(QuickTime/mp4), 프레임 범위, 상태 등
- **PublishedFile** — 파이프라인 산출 파일(.ma, .nk, .exr 등)의 메타데이터
- **Note** — 코멘트/피드백. 버전·샷·태스크 등 어디에나 붙을 수 있음. 스레드형 답글.
- **Playlist** — 리뷰 세션에서 함께 볼 Version들의 묶음(데일리, dailies)

### 2.5 횡단 개념
- **Status (sg_status_list)** — 상태값. 색상 코드와 함께. 예:
  - `wtg`(Waiting to Start), `rdy`(Ready), `ip`(In Progress),
    `rev`(Pending Review), `fin`(Final), `hld`(On Hold), `omt`(Omit), `apr`(Approved)
- **Custom Fields** — 모든 엔티티에 사용자 정의 필드 추가 가능 (FPTR의 핵심 유연성)
- **Filters / Saved Pages** — 그리드의 필터·정렬·그룹핑을 저장한 뷰

## 3. 엔티티 관계 (요약)

```
Project 1───* Sequence 1───* Shot
Project 1───* Asset
Shot   *───* Asset           (샷에 등장하는 에셋: 다대다)
Shot   1───* Task            (샷의 작업들)
Asset  1───* Task
Task   *───1 Step            (작업 공정 단계)
Task   *───* HumanUser       (담당자: 다대다)
Task   1───* Version         (작업의 산출 버전들)
Version *───1 HumanUser      (제출자)
Note   *───* (anything)      (다형 연결: 버전/샷/태스크/에셋에 붙음)
Playlist *───* Version
```

이 구조의 핵심은 **Task가 Shot/Asset 모두에 동일한 방식으로 붙는다**는 점과,
**Note가 어떤 엔티티에도 붙는 다형(polymorphic) 관계**라는 점이다.

## 4. 핵심 UI / 화면 패턴

FPTR UI는 본질적으로 **"필터 가능한 거대한 스프레드시트 + 미디어 플레이어"** 다.

1. **Project 대시보드** — 프로젝트 진척, 내 할 일(My Tasks)
2. **엔티티 그리드(Entity Grid)** — Shot/Asset/Task를 표(엑셀형)로 표시.
   - 인라인 편집, 컬럼 커스터마이즈, 필터, 그룹핑, 정렬
   - 썸네일 모드 / 리스트 모드 전환
3. **Detail 페이지** — 한 엔티티의 모든 정보 + 연관 탭(Tasks, Versions, Notes)
4. **My Tasks** — 로그인 사용자에게 할당된 태스크 목록
5. **Media / Review** — Version 썸네일 그리드, 플레이어, 노트 작성
6. **Schedule / Gantt** — 태스크 일정을 간트 차트로

> 핵심 인터랙션: **인라인 셀 편집으로 상태(Status)를 바로 바꾸는 것**. 아티스트와
> 코디네이터가 하루 수십 번 하는 동작이라 프로토타입에서도 반드시 재현 대상.

## 5. 통합/확장 (프로토타입 범위 밖이나 기록)

원본 FPTR의 강점은 단순 추적이 아니라 **DCC 툴 파이프라인 통합**이다.
- **Toolkit (tk / sgtk)** — Maya/Nuke/Houdini에 붙는 파이프라인 프레임워크
- **Python API (shotgun_api3)** — REST 위에 올라간 공식 파이썬 클라이언트
- **Event Daemon / Webhooks** — 변경 이벤트 기반 자동화
- **RV / Screening Room** — 전문 리뷰 플레이어

→ 프로토타입은 **웹 UI + REST API + DB 도메인 모델**에 집중하고, DCC 통합은 제외한다.
   다만 REST API는 shotgun_api3와 유사한 **엔티티 CRUD 패턴**을 따라 설계하여
   "기능적 동등성"의 방향성을 보인다. (상세 `05_API설계.md`)

## 6. 프로토타입 재현 우선순위 결론

| 우선순위 | 기능 | 근거 |
|---------|------|------|
| ★★★ 필수 | Project / Shot / Asset / Task / Version / Note 엔티티 CRUD | 도메인의 뼈대 |
| ★★★ 필수 | 엔티티 그리드 + 인라인 상태 편집 | FPTR의 핵심 일상 인터랙션 |
| ★★★ 필수 | Task 담당자 할당 + Status 워크플로우 | Track의 본질 |
| ★★☆ 중요 | Version 미디어 + Note 리뷰 | Review의 본질 |
| ★★☆ 중요 | My Tasks / 대시보드 | 사용자 진입점 |
| ★☆☆ 선택 | Playlist, Gantt, Custom Field UI | 시간 허락 시 |
| ☆☆☆ 제외 | DCC 통합, 권한관리(RBAC), 실시간 협업 | 프로토타입 범위 밖 |

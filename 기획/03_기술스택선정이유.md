# 03. 기술 스택 선정 이유

원본과의 기능적 동등성을 검증하면서, 사용자 제약(로컬 격리, 설치형 DB 금지)을
만족하는 스택을 선정했다.

---

## 1. 최종 선정 스택

| 영역 | 선택 | 격리 방식 |
|------|------|-----------|
| 백엔드 언어 | **Python 3.9** | `개발/backend/venv` 로컬 가상환경 |
| 백엔드 프레임워크 | **FastAPI** + Uvicorn | venv 내 설치 |
| ORM | **SQLAlchemy 2.x** | venv 내 설치 |
| 검증/직렬화 | **Pydantic v2** | venv 내 설치 |
| 데이터베이스 | **SQLite** (파일 DB) | `개발/database/ftpr.db` |
| 프론트 언어 | **TypeScript** | `개발/frontend/node_modules` 로컬 |
| 프론트 프레임워크 | **React 18** | 로컬 설치 |
| 빌드/개발서버 | **Vite** | 로컬 설치 |
| 라우팅 | **React Router** | 로컬 설치 |
| 서버 상태 | **TanStack Query** | 로컬 설치 |

---

## 2. 선정 근거 (Why)

### 2.1 왜 Python 백엔드인가
- **원본 FPTR의 공식 API(`shotgun_api3`)와 파이프라인 Toolkit이 모두 Python**이다.
  VFX 업계의 사실상 표준 언어. "기능적 동등성"을 논할 때 같은 언어 생태계에 있는 것이
  설득력이 크다.
- FastAPI: 자동 OpenAPI(Swagger) 문서, Pydantic 기반 검증, 비동기 지원.
  → 엔티티 CRUD가 많은 본 도메인에 적합하고, `/docs`로 API 탐색이 즉시 가능.
- 대안 검토:
  - Django + DRF → 무겁고 admin이 강하지만 프로토타입엔 과함.
  - Node/Express → 프론트와 언어 통일 장점 있으나 VFX 표준성에서 Python이 우위.

### 2.2 왜 SQLite인가 (사용자 제약 직접 반영)
- 사용자 요구: **설치형 DB 금지 → 파일 DB**.
- SQLite는 별도 서버 프로세스/설치가 전혀 없고 단일 `.db` 파일로 동작.
- Python 표준 라이브러리에 `sqlite3` 내장 → 추가 설치 부담도 최소.
- 프로토타입 규모(수천 행)에서 성능 충분.
- 한계: 동시 쓰기 약함, 일부 ALTER 제약 → 프로토타입에선 문제 없음.
  (운영 전환 시 PostgreSQL로 교체 가능하도록 SQLAlchemy ORM으로 추상화)

### 2.3 왜 React + TypeScript + Vite인가
- 원본 FPTR UI의 핵심은 **인라인 편집되는 거대한 데이터 그리드**.
  컴포넌트 단위 상태관리가 강한 React가 적합.
- TypeScript: 엔티티 필드가 많아 **타입 안정성**이 유지보수에 결정적.
  사용자 제약(폴더 내 격리)도 `node_modules` 로컬 설치로 자연히 충족.
- Vite: 설정 최소, 빠른 HMR. 프로토타입 반복 개발에 최적.
- TanStack Query: 서버 상태 캐싱/리패칭으로 그리드의 실시간성 흉내내기 쉬움.

### 2.4 왜 프론트/백 분리(REST)인가
- 사용자 제약: **언어 혼용 시 명시적 분리**.
  → `backend`(Python) / `frontend`(TypeScript)를 디렉토리로 갈라 REST로만 통신.
- 원본 FPTR도 본질적으로 **REST API + 웹 클라이언트** 구조 → 아키텍처 동질성.

---

## 3. 격리(Isolation) 보장 방법

| 대상 | 보장 |
|------|------|
| Python 패키지 | `python3 -m venv 개발/backend/venv` 후 그 venv의 pip로만 설치. global pip 미사용. |
| TS 패키지 | `개발/frontend`에서 `npm install` → `node_modules` 로컬 생성. `-g` 미사용. |
| DB | 파일 1개(`개발/database/ftpr.db`). 시스템 서비스 등록 없음. |
| 실행 | `실행방법.md`의 스크립트가 venv/node를 로컬 경로로만 호출. |

---

## 4. 디렉토리-언어 매핑 (명시적 분리)

```
개발/
├── backend/    → Python 전용 (venv 포함). .py 파일만.
├── frontend/   → TypeScript 전용 (node_modules 포함). .ts/.tsx 파일만.
└── database/   → 언어 중립. .db 파일만.
```

이 경계는 물리적으로 강제되며, 두 언어가 한 디렉토리에 섞이지 않는다.
